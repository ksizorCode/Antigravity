📌1. Resumen ejecutivo

TuristSpain nace para resolver una fricción concreta: cuando una persona está de viaje o en una escapada, quiere saber rápido qué merece la pena visitar cerca, dónde comer, qué actividades tiene a mano y cómo reservar sin perder media mañana comparando fuentes dispersas.

El proyecto se plantea como una Progressive Web App orientada a uso móvil. El usuario permite la ubicación, el sistema calcula la distancia respecto a los puntos de interés almacenados y devuelve recomendaciones ordenadas por cercanía, categoría y utilidad. El producto evita dos errores habituales en apps turísticas: mostrar demasiadas opciones irrelevantes y depender de búsquedas manuales largas.

El MVP se ha desarrollado mediante una metodología de vibe coding, apoyada por asistentes de IA como Google Antigravity. El rol humano no desaparece: se desplaza desde la escritura manual de cada línea hacia la dirección de producto, la definición de requisitos, la validación del resultado, la depuración funcional y la toma de decisiones sobre experiencia de usuario.

Idea fuerza

TuristSpain convierte una pregunta ambigua —“¿qué hago cerca de aquí?”— en una respuesta práctica: lugares cercanos, datos accionables, distancia aproximada, horarios, teléfono, mapa y enlace de reserva.

🌍2. Contexto y oportunidad

El turismo español tiene una paradoja clara: dispone de una enorme riqueza patrimonial, gastronómica, natural y local, pero el usuario medio termina visitando siempre los mismos puntos visibles en grandes plataformas. Esto produce saturación en unos pocos lugares y baja visibilidad para municipios, restaurantes, rutas y actividades de alto valor que no cuentan con una estrategia digital potente.

TuristSpain ataca esa asimetría. No intenta competir como una red social turística generalista ni como un marketplace gigante. Su posicionamiento es más preciso: descubrimiento de proximidad accionable. El valor no está en tener millones de registros, sino en mostrar pocos resultados útiles en el momento exacto.

🏘️

Economía local

Da visibilidad a negocios, recursos rurales y experiencias que no aparecen en los primeros resultados de buscadores.

🧠

Menos saturación mental

Reduce la comparación interminable entre blogs, mapas, reseñas, redes sociales y webs oficiales.

🧭

Decisión contextual

⚠️3. Problema turístico

3.1 Problema principal

El usuario no carece de información. Carece de información filtrada, fiable y accionable. La abundancia de fuentes genera una experiencia lenta: abrir Google Maps, mirar reseñas, comparar fotos, buscar horarios, comprobar si hay reserva, validar distancia real y repetir el proceso para cada alternativa.

Este coste de búsqueda perjudica especialmente a turistas con tiempo limitado, familias, personas mayores, viajeros en coche, agencias que preparan itinerarios y municipios que dependen de que el visitante encuentre actividades complementarias.

🧩7. UX y flujos de interacción

La experiencia se diseña con un principio: mínima fricción, máxima utilidad. El usuario no debe aprender un sistema complejo. Debe reconocer rápidamente dónde está, qué hay cerca y qué acción puede ejecutar.

7.1 User journey principal

1

Entrada

El usuario abre la PWA desde navegador o icono instalado. Se carga la interfaz base y se explica el valor de activar la ubicación.

2

Permiso de ubicación

La app solicita GPS solo cuando es necesario. Si el usuario rechaza el permiso, debe poder buscar manualmente por zona en una fase futura.

3

Consulta de datos

El frontend obtiene los POIs desde Supabase y aplica cálculo de distancia. Los resultados se filtran por radio y categoría.

4

Exploración

El usuario alterna entre modo lista, tabla o ficha visual según su necesidad.

5

Acción

La decisión se convierte en llamada, reserva, navegación en mapa o visita a la web oficial.

19.1 Lectura estratégica del mercado

🧬22. Anexo D · Esquema SQL propuesto para Supabase

Este anexo añade una propuesta técnica concreta de base de datos. No sustituye la implementación real, pero sirve como referencia para desplegar el MVP con más rigor y explicar cómo se gobernarían permisos y calidad de datos.

22.1 Tabla principal

create table if not exists public.pois ( id uuid primary key default gen_random_uuid(), name text not null, description text, category text not null check (category in ('cultura','naturaleza','gastronomia','actividad','alojamiento','otro')), latitude numeric(9,6) not null check (latitude between -90 and 90), longitude numeric(9,6) not null check (longitude between -180 and 180), schedule text, phone text, website_url text, booking_url text, image_url text, municipality text, province text, autonomous_region text, status text not null default 'active' check (status in ('active','draft','archived','needs_review')), source_url text, created_at timestamptz not null default now(), updated_at timestamptz not null default now() ); create index if not exists pois_category_idx on public.pois(category); create index if not exists pois_status_idx on public.pois(status); create index if not exists pois_location_idx on public.pois(latitude, longitude);

22.2 Políticas de seguridad recomendadas

alter table public.pois enable row level security; create policy "Public read active POIs" on public.pois for select using (status = 'active'); create policy "Authenticated editors can insert" on public.pois for insert to authenticated with check (true); create policy "Authenticated editors can update" on public.pois for update to authenticated using (true) with check (true);

Nota de seguridad

En un producto real, la política de escritura no debería abrirse a cualquier usuario autenticado. Debería limitarse a un rol editor o administrador mediante claims, tabla de perfiles o funciones RPC.

22.3 Función futura para distancia en base de datos

En el MVP el cálculo puede vivir en cliente. Si el dataset crece, conviene mover el filtrado de distancia al servidor o usar PostGIS. Esto reduce transferencia de datos y mejora rendimiento.

-- Evolución recomendada si el volumen de POIs crece: -- 1. Activar extensión PostGIS. -- 2. Añadir columna geography(Point, 4326). -- 3. Indexar con GIST. -- 4. Consultar por ST_DWithin para devolver solo POIs cercanos.

🧯35. Anexo P · Limitaciones actuales y supuestos

Un proyecto serio debe separar lo que está construido, lo que está diseñado y lo que aún es hipótesis. Esta separación evita exagerar el alcance y facilita que el lector entienda el estado real del MVP.

35.1 Limitaciones del MVP

  • La cobertura depende del dataset cargado. Si una zona no tiene POIs, la app no puede inventar recomendaciones.
  • El cálculo de distancia en cliente es suficiente para un prototipo, pero no es la mejor solución para una base de datos grande.
  • Los horarios pueden ser complejos: festivos, temporadas, cierres parciales o cambios de última hora requieren un modelo más avanzado.
  • La geolocalización depende del dispositivo, navegador, permisos y precisión disponible en ese momento.
  • El MVP no incorpora sistema propio de reservas, pagos ni reseñas; deriva al usuario a canales externos.
  • La personalización todavía no aprende del comportamiento del usuario, por privacidad y falta de volumen de datos.

35.2 Supuestos de trabajo

El proyecto asume que el usuario está dispuesto a activar la ubicación si entiende el beneficio, que el radio de 75 km es útil para escapadas y rutas cortas, y que la categoría gastronómica tiene especial capacidad de conversión. También asume que una parte del valor puede venir de usuarios profesionales, no solo de turistas finales.

Estos supuestos son razonables para un MVP, pero deben validarse. La forma correcta de hacerlo es medir uso real: tasa de permiso GPS, tiempo hasta primera acción, clics en mapa, llamadas, reservas y feedback sobre calidad de resultados. Si los datos contradicen una hipótesis, el producto debe ajustarse. Por ejemplo, si muchos usuarios rechazan ubicación, la búsqueda manual debe subir de prioridad. Si la tabla la usan solo agencias, puede mantenerse como función profesional y simplificarse para turistas.

35.3 Decisiones que no deben cambiarse sin evidencia

No conviene ampliar categorías de forma impulsiva, añadir una red social, introducir pagos o crear recomendaciones opacas antes de validar el flujo básico. Cada función adicional aumenta mantenimiento, riesgo legal, complejidad de datos y carga cognitiva. La dirección recomendada es mejorar calidad, cobertura y confianza antes que acumular funcionalidades.

35.4 Recomendación final de gestión

La recomendación final es mantener el proyecto bajo una lógica de evidencia. Primero se comprueba que el usuario entiende la propuesta, encuentra resultados cercanos y pulsa una acción útil. Después se amplía cobertura territorial. Solo cuando exista uso repetido tiene sentido añadir automatizaciones, personalización o monetización. Este orden protege el proyecto de uno de los errores más comunes en productos digitales: construir muchas funciones antes de demostrar que el flujo principal resuelve un problema real. En TuristSpain, el flujo principal es claro y debe seguir siendo el centro: ubicación, resultados relevantes, comparación sencilla y acción inmediata.

También conviene conservar el historial de prompts, decisiones y cambios realizados con IA. Esa trazabilidad permite explicar qué se pidió, qué devolvió la herramienta, qué se aceptó, qué se rechazó y qué se corrigió manualmente. En un contexto académico o profesional, esta práctica aumenta la credibilidad del proyecto y demuestra que la IA se usó como apoyo metodológico, no como sustituto del criterio técnico.

Como cierre operativo, el proyecto debe revisarse con una demo real en móvil, una explicación breve del flujo y una muestra de datos suficientemente variada. La defensa será más convincente si se enseña el problema antes que la tecnología: primero la pérdida de tiempo del turista, después la solución de proximidad y finalmente la arquitectura que la hace posible.

Este enfoque mantiene la memoria alineada con una evaluación profesional: claridad del problema, evidencia del prototipo, límites reconocidos, metodología de IA documentada y próximos pasos realistas.

La prioridad inmediata es probar, medir, corregir y ampliar datos sin perder simplicidad.

Ese criterio debe mantenerse durante todo el desarrollo posterior.
